## Narrative

**(a) Monorepo vs per-repo split.** The MCP ecosystem is bimodal. A handful of large, organization-backed monorepos (`modelcontextprotocol/servers`, `awslabs/mcp`, `microsoft/mcp`, `googleapis/mcp-toolbox`) use **language-native workspace conventions** (npm workspaces, uv/pyproject, .NET slnx) — each server lives at `src/<name>/` or `servers/<name>/` and is an independent publishable package. The vast majority of MCP servers, however, are **single-server, single-repo**, often a thin wrapper that publishes to npm/PyPI/Docker. There is essentially no middle ground: nobody ships 3–5 related servers in one repo without using a workspace tool. The monorepo cases all share one tell — a root `.mcp.json` for *dev-time* server config, distinct from per-server `server.json` install manifests.

**(b) Aggregators handle registry-of-registries problem three ways.** (1) **Pure README** — `punkpeye/awesome-mcp-servers` (87K stars, 737KB markdown), `appcypher`, `wong2`. No machine-readable index; humans browse, humans copy install snippets. (2) **Issue-driven curation** — `cline/mcp-marketplace` accepts submissions via GitHub Issues and relies on Cline to read each server's `README.md` + optional `llms-install.md` to perform install. (3) **Structured registry** — `modelcontextprotocol/registry` (one `server.json` per server, federated by reverse-DNS name), `docker/mcp-registry` (per-server `server.yaml` directories with Docker-image + secrets schema). Only the third class is machine-consumable; #1 dominates by stars but is opaque to tooling.

**(c) Canonical install metadata.** The MCP-registry `server.json` schema (`https://static.modelcontextprotocol.io/schemas/2025-12-11/server.schema.json`) is consolidating: reverse-DNS `name`, `version`, `repository`, and a `packages[]` array where each entry has `registryType` (npm | oci | pypi | mcpb), `identifier`, `transport` (stdio/http/sse), `runtimeHint`, `environmentVariables[]` (with `isSecret`, `isRequired`, `format`, `choices`, `default`), and optional `fileSha256`. The `package.json` `mcpName` field is the bridge from npm to registry id. Competing/complementary formats: **smithery.yaml** (JSON-Schema user config + `commandFunction`), **mcpb manifest** (`manifest_version: 0.3` with full `tools[]` enumeration and embedded `mcp_config`), **gemini-extension.json**, **glama.json**, and **the original Claude Desktop `mcpServers` map** (which is *runtime config*, not a publish manifest). Many high-star servers ship 4–6 of these in parallel.

**(d) Signal for ACIF `mcp_config`.** The single sharpest finding: the MCP world distinguishes **install/publish manifest** (`server.json` — describes how to acquire and launch) from **runtime/user config** (`.mcp.json` — what *this user* has wired up). ACIF's `mcp_config` kind is naturally the second one — closer to the `.mcp.json` `mcpServers` map than to `server.json`. The first is owned by `modelcontextprotocol/registry` and its schema. If ACIF treats `mcp_config` as "the user's wiring of an MCP server into a tool," then it should reference upstream `server.json` by registry id (`io.github.<owner>/<name>`) rather than re-spec install metadata.

## ACIF Implications

- **`mcp_config` is configuration, not content.** Treat it as a thin pack-level wiring artifact (server name + selected transport + user-supplied env binding) that references an external `server.json` by `mcpName`/registry id. Do not redefine `packages[]`, `runtimeHint`, or `environmentVariables[]` — defer to the MCP registry schema.
- **Two distinct manifests must both be recognized.** `server.json` (publish/install side, owned by MCP registry) and `.mcp.json` (runtime `mcpServers` map, owned by Claude Code / Desktop). ACIF should specify which one a pack carries — and the survey suggests *both can coexist in one repo*.
- **Appendix A precedence:** `.mcp.json` at repo root is the established convention for runtime wiring (seen in `modelcontextprotocol/servers`, `mcp-use`, `XcodeBuildMCP`, `archestra`). If ACIF uses Appendix A precedence, `.mcp.json` should map to `mcp_config` kind and live near the top of the precedence chain. `server.json` should map to a separate `mcp_server` install manifest (or be explicitly out of ACIF's scope).
- **Cross-kind co-shipping is real.** `XcodeBuildMCP` and `unity-mcp` ship `skills/` alongside MCP server code. ACIF's pack model already handles this — confirms the carrier-model decision to let one pack contain multiple kinds.
- **Filename plurality is the norm, not the exception.** DesktopCommanderMCP carries six install/discovery manifests. ACIF must not assume a single canonical manifest at the repo root — sidecar-by-kind is correct.
- **Reverse-DNS naming is dominant** in the registry world (`io.github.<owner>/<repo>`). ACIF item identity for `mcp_config` items should align with this, not invent a parallel namespace.
- **Aggregators (87K-star tier) are README-only.** ACIF will not displace them; the realistic integration is "ACIF pack opt-in" for individual server repos, with aggregators continuing to point at upstream READMEs. Don't design for the aggregator use case.
- **Open question for the spec:** does ACIF carry the *install manifest* (publish-side) or only the *user wiring* (config-side)? The survey strongly suggests **only the wiring**. The MCP registry already solves install metadata, and duplicating it would put ACIF in conflict with `server.json` rather than complementary to it.
