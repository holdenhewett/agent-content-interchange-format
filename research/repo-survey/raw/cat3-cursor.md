## Narrative

**The `.cursorrules` → `.mdc` transition is mid-flight.** Cursor's original convention was a single `.cursorrules` file at the repo root — a plain-text system prompt with no structure (BlueBirdBack, grapeot, kinopeee). The new convention is `.cursor/rules/*.mdc`: one rule per file with YAML frontmatter and Cursor-specific keys (`description`, `globs`, `alwaysApply`). Every repo created or updated after early 2025 ships `.mdc`. PatrickJS preserves both eras side-by-side (`rules/` legacy, `rules-new/` mdc), and Sanjeed5 explicitly labels the old layout `rules-v0-deprecated/`. The frontmatter vocabulary has converged on three fields — `description`, `globs`, `alwaysApply` — across 11 of 18 repos. This is a strong signal for **ACIF Decision #10**: where frontmatter exists, publishers already accept it, and the keys are narrow and stable. ACIF's `description` field maps 1:1.

**By-topic vs by-language organization in aggregators.** PatrickJS chose flat-slug dirs (`nextjs-typescript-cursorrules-prompt-file/`) — searchable but messy. Sanjeed5 flattened further to single-file `.mdc` per library and overlays a `rules.json` catalog with tags. Aaronontheweb organizes by **technology layer** (csharp/, dotnet-sdk/, nuget-packages/, ci-cd/) with a README per topic. Jabrena uses **numbered prefixes** as a kind/phase taxonomy (012-agile-epic, 030-architecture-adr, 110-java-maven). Three distinct organizing axes — slug, tag, prefix — and Cursor itself imposes none of them. The directory layout is purely social.

**How aggregators normalize.** Sanjeed5's `rules.json` is the most explicit example: it's a sidecar registry mapping each `.mdc` file to `{name, tags[]}`. The `.mdc` frontmatter handles per-rule self-description; the JSON catalog handles **cross-rule discovery and faceting** — exactly the split ACIF proposes (per-item frontmatter is optional, sidecar registry is canonical). instructa/ai-prompts' `aiprompt.json` does similar work with explicit `files`, `version`, `ai_editor`, and `tech_stack` keys — a vendor-neutral sidecar manifest.

**Multi-file packs.** Two strong patterns: (a) PatrickJS dir-bundles where a slug dir holds `.cursorrules` + multiple `.mdc` siblings + README, discovered by relative-path implicit; (b) grapeot/agentic-cursorrules where the rule references sibling `tools/` or `scripts/` dirs by convention — no manifest, brittle.

## ACIF implications

- **Frontmatter convergence on 3 keys** (`description`, `globs`, `alwaysApply`) confirms ACIF's Decision #15 (frontmatter opt-in, narrow) — the ecosystem has organically converged on a tiny vocabulary; ACIF needs only to claim a non-colliding namespace (e.g. `acif.*`).
- **Sidecar registry pattern is already proven** by Sanjeed5's `rules.json` and instructa's `aiprompt.json` — neither requires modifying the rule files themselves. Validates ACIF's "sidecar universal" stance.
- **Carrier flexibility is non-negotiable.** 18 repos exhibit 5 distinct layout patterns (flat-root, flat-`.cursor/rules`, by-type, by-topic, dir-bundles). Any ACIF spec that mandates a layout will be rejected by half the ecosystem; the carrier must be layout-agnostic.
- **Mixed content kinds in one repo are common.** Jabrena ships skills + agents + commands + mcp.json + rules under `.cursor/`. ACIF must handle heterogeneous packs natively, not assume one repo = one content type.
- **Aux-file discovery is the weakest seam.** Most repos use relative-path-implicit (rule references sibling files by convention). ACIF's explicit `files[]` (or equivalent) is a genuine improvement, but adoption will lag without tooling.
- **Versioning is almost absent.** Per-item versions are rare; most rely on git-tags or nothing. ACIF should make per-item version optional.
- **The `.mdc` extension is Cursor-specific** and blefnk shows the alternative (`.md` with Cursor frontmatter for cross-tool reuse). ACIF should not require any particular extension — content-type detection must accept both.
