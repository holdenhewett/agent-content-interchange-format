## Narrative

The non-Claude AI tool ecosystem has fractured into roughly four conventions that ACIF must reckon with.

**1. The dotfile-dir pattern (Cline, Roo, Continue, Goose).** Each tool claims a top-level dotfile directory: `.clinerules/`, `.roo/`, `.continue/`, `.goose/`. Inside, content is sub-typed (`rules/`, `agents/`, `prompts/`, `checks/`, `skills/`). Most rule files are Markdown with optional YAML frontmatter (`description`, `globs`, `applyTo`, `tags`). There is no pack manifest — discovery is purely by directory convention. Cline and Roo also produce *tool config* files alongside (`.roomodes`, `.goosehints`) that are YAML and machine-parsed. The cline/cline repo itself ships eight parallel dotfile directories (`.clinerules`, `.claude`, `.codex`, `.agents`, etc.), which is a vivid demonstration of the multi-tool tax authors are paying today.

**2. The single-file AGENTS.md movement (agentsmd, openai/codex, OpenCLI, SwiftAgents, ciembor).** This is a deliberate counter-trend: "stop sprouting dotfile dirs, put everything in `AGENTS.md` at the repo root." The agentsmd/agents.md spec is intentionally minimal — "it's just Markdown, put it at the root." Codex, Cursor, Gemini CLI, Amp, and OpenCode all read it. As of this survey it's the closest thing to a cross-tool de-facto standard, and ACIF would be wise not to fight it.

**3. The structured-config pattern (Aider, genaiscript, instructa).** Aider's `.aider.conf.yml` declares a `read:` list of files to inject — a real pack manifest. instructa/ai-prompts ships an `aiprompt.json` sidecar with `files`, `ai_editor`, `tech_stack`, `version` — strikingly close to ACIF's own carrier model. genaiscript embeds the manifest inside a TypeScript `script({...})` call.

**4. The cookbook/recipe pattern (Goose, github/awesome-copilot).** Goose recipes are `recipe.yaml` with `parameters`, `activities`, `settings`, `instructions` — a templated YAML format. awesome-copilot ships a `cookbook.yml` index plus JSON Schemas (`collection.schema.json`, `cookbook.schema.json`, `tools.schema.json`) — the most ACIF-aligned manifest discipline in the survey.

The Microsoft/skills and awesome-copilot repos explicitly adopt Anthropic's `SKILL.md` + frontmatter shape, and the Copilot docs literally instruct authors to put skills in either `.github/skills/` or `.claude/skills/`. That convergence on `SKILL.md` is real, but it does NOT extend to rules, agents, hooks, or MCP configs, where every tool still does its own thing.

## ACIF Implications

- **AGENTS.md is the dominant cross-tool rule format.** ACIF's carrier model should treat repo-root `AGENTS.md` as a first-class carrier surface — likely as a sidecar target or a "render to AGENTS.md" emit option — not just a Claude/CLAUDE.md variant. Ignoring it isolates ACIF from the broadest existing cross-tool standard.
- **SKILL.md frontmatter is already the de-facto skill format outside Claude too** (Copilot, Microsoft/skills, Roo, Continue, OpenCode all use it). ACIF should normatively align its `skill` content type with this shape — it's the one shared vocabulary that already exists.
- **Most tool-specific rule dirs (`.clinerules/`, `.roo/`, `.continue/`, `.goose/`, `.cursor/`) have no pack manifest.** They rely on directory convention. ACIF's universal sidecar gives publishers something they currently lack, but uptake requires ACIF to also explain how to *emit* into these convention-only dirs.
- **The "multi-tool tax" is real and visible.** cline/cline ships 8 parallel dotfile dirs; goose ships 7+; claude-task-master generates installers for 6 tools. This is exactly the pain ACIF promises to fix — frame the value prop as "publish once, render to N tools."
- **Pack-manifest precedents exist and are converging on similar fields.** awesome-copilot's `cookbook.yml` + JSON schemas, instructa's `aiprompt.json` (`files`/`version`/`ai_editor`/`tech_stack`), and Aider's `.aider.conf.yml` (`read:` list) all encode roughly: id, version, file-list, tool targeting, tags. ACIF's registry schema should explicitly map to these so adapters are mechanical.
- **Non-markdown content units exist (genaiscript `.genai.mts`, Goose `recipe.yaml`, Roo `.roomodes`).** ACIF's body_hash and carrier model must not assume Markdown bodies — YAML, TypeScript, and templated YAML (Jinja-style placeholders in Goose) are all real cases.
- **Hooks are diverging fast.** Microsoft/skills uses Anthropic's `hooks.json` (sessionStart/postToolUse/sessionEnd) verbatim; Cline has its own `hooks/` dir; Continue calls them "checks." If ACIF wants a portable `hook` content type it will have to pick a shape and accept that adapters will translate — there's no convergence yet to ride.
- **Multi-size variants are a real pattern (ciembor's `.md`/`.mini.md`/`.nano.md`).** Authors are manually solving for token budgets. ACIF should consider whether `variants` or `size_tier` belongs in the manifest, or stay out and let publishers ship multiple items — but the use case is documented in the wild.
